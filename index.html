What is the ESP32?

The ESP32 is a dual-core 32-bit microcontroller made by Espressif Systems. It runs at up to 240 MHz, has 520 KB of SRAM, and integrates Wi-Fi and Bluetooth radios on the same die. For this project, the key capability is its interrupt system: any GPIO pin can be configured to call a function instantly when it detects a rising or falling edge, without the main code needing to poll it.

The ISR (Interrupt Service Routine) used for pulse counting is placed in IRAM — a section of fast on-chip memory — so it executes within a few nanoseconds of the edge event, ensuring no pulses are ever missed even at high shaft speeds.

// ESP32 IRAM-resident ISR — executes in fast memory void IRAM_ATTR encoderISR() { pulseCount++; dirForward = (digitalRead(ENCODER_DT) == HIGH); }

Advantages
  • Extremely fast GPIO interrupt response makes high-PPR encoders viable without missing pulses
  • Dual-core architecture means a heavy display update cannot block the counting ISR
  • Built-in Wi-Fi and Bluetooth are available for free if you want to stream RPM data over a network
  • Abundant community resources, libraries, and tutorials
  • Very affordable — under $5 for a complete development board
  • Deep sleep modes allow battery-powered installations to last much longer
Disadvantages
  • GPIO 6–11 are internally connected to the flash chip and cannot be used for general I/O
  • All GPIO operate at 3.3 V logic; 5 V signals from older encoder modules need a level shifter
  • ADC accuracy is poor when Wi-Fi is active (not relevant here, but worth knowing)
  • No hardware floating-point division — RPM formula uses a tiny amount of software FPU time
  • Higher idle power draw compared to AVR-based boards if battery life matters

How a rotary encoder works

An incremental rotary encoder converts shaft rotation into a sequence of electrical pulses. The most common low-cost variant used in hobby projects is the optical slot encoder: a disc with evenly-spaced holes or slots rotates between an infrared LED and a photodetector. Each time a hole passes the beam, the photodetector output switches state, generating one pulse. The number of pulses per revolution is fixed by the number of slots in the disc (PPR).

Quadrature encoders produce two channels — A and B — 90 degrees out of phase with each other. By checking which channel leads the other at the moment of a rising edge, direction of rotation can be determined unambiguously. This project reads direction from channel B at each rising edge of channel A.

Advantages
  • Non-contact sensing — the disc never touches the sensor, giving very long mechanical life
  • Absolute repeatability — each slot is the same width, so PPR is constant across the entire speed range
  • Direction detection with just two pins using the quadrature technique
  • No calibration needed once PPR is set in firmware
  • Low-cost slot disc modules are available for under $2
Disadvantages
  • Minimum measurable RPM is limited by the sample window — at very low speeds, few pulses arrive in 200 ms
  • Slot discs are susceptible to dust and oil contamination blocking the IR beam
  • Contact encoders (like KY-040) exhibit mechanical bounce, requiring debounce logic or filtering
  • Resolution is fixed by disc geometry — you cannot increase PPR without changing the hardware
  • Mis-alignment of the disc on the shaft introduces periodic errors in the pulse spacing

SSD1306 OLED display

The SSD1306 is a single-chip CMOS OLED driver controller produced by Solomon Systech. It manages a 128x64 matrix of organic light-emitting pixels. Unlike LCD panels, OLED pixels emit their own light, so there is no backlight. Black pixels consume no power at all, and white pixels produce a bright, high-contrast image visible in direct sunlight or darkness alike.

Communication happens over I2C using just two wires (SDA and SCL). The Adafruit SSD1306 library maintains a 1 KB frame buffer in the ESP32's SRAM. Drawing calls (text, lines, rectangles) modify this buffer. Calling display.display() pushes the entire buffer to the controller over I2C in one shot, which at 400 kHz I2C clock takes about 3 milliseconds.

// Buffer → display transfer happens here void drawRPMScreen(float rpmVal) { display.clearDisplay(); // wipe buffer (fast, in SRAM) display.setTextSize(3); display.setCursor(30, 16); display.print((int)rpmVal); // write to buffer display.display(); // push buffer over I2C (~3 ms) }

Advantages
  • Excellent contrast ratio — black pixels are truly off, making text sharply readable

  • Only two wires needed for the entire display (I2C)
  • Very thin and lightweight at roughly 1 mm panel thickness
  • The Adafruit library is mature, stable, and well documented
  • No backlight means power consumption scales directly with the number of lit pixels
Disadvantages
  • OLED pixels degrade over time — pixels that are always on burn in faster than those that cycle
  • I2C limits update rate — full-frame refresh at 400 kHz takes ~3 ms, meaning a maximum of ~330 frames per second theoretically, but library overhead brings this lower in practice
  • 128x64 resolution is limiting for graphical content; suitable for text and simple gauges only
  • Blue OLED variants have a noticeably shorter lifespan than white variants
  • Very small physical size (0.96 inch) makes it difficult to read from more than 50 cm away

Stage 1 — Physical rotation

The encoder disc is mounted directly on the rotating shaft. As it turns, the slots in the disc alternately block and unblock the infrared beam between the LED and photodetector. Each unblocked event (rising edge) produces one voltage transition from low to high on the CLK output pin.

Stage 2 — Pulse generation and interrupt capture

The CLK pin is connected to GPIO 34 of the ESP32, which is configured with attachInterrupt(RISING). The hardware interrupt controller detects the voltage edge and immediately pauses the main CPU context, jumps to the ISR in IRAM, increments the counter, and resumes normal code execution. The entire interrupt takes under 200 nanoseconds. The DT (Channel B) pin is read at this instant to record direction.

Stage 3 — Time-windowed counting

The main loop checks millis() against the last sample timestamp. Every 200 ms (configurable via SAMPLE_MS) it pauses interrupts for a few microseconds, copies the pulse counter to a local variable, resets the counter to zero, and re-enables interrupts. This snapshot approach ensures the counter is never read while the ISR is mid-increment.

Stage 4 — RPM calculation and filtering

The formula converts pulses to RPM:

// Core formula RPM = (count / PPR) * (60000 / SAMPLE_MS) // count = pulses measured in this window // PPR = pulses per revolution (from encoder spec) // SAMPLE_MS = window length in milliseconds // 60000 = milliseconds per minute // Exponential moving average (smoothing) rpmSmoothed = ALPHA * rawRPM + (1.0 - ALPHA) * rpmSmoothed;

The exponential moving average filter blends each new raw reading with the previous filtered value. With ALPHA = 0.3, the filter reaches ~95% of a step change in about 8 sample windows (1.6 seconds), which feels responsive while eliminating single-sample spikes.

Stage 5 — Display refresh

The Adafruit SSD1306 library maintains a 1024-byte (128*64/8) monochrome frame buffer in SRAM. Each call to clearDisplay(), print(), drawRect(), etc. modifies this buffer instantly without touching the display hardware. Only when display.display() is called does the library transfer the entire buffer to the SSD1306 over I2C at 400 kHz. This batched transfer takes about 3 ms and updates the screen in one clean frame with no partial-update flicker.

The OLED shows: a large three-digit RPM value, a direction indicator (CW / CCW), a running total revolution counter, and a horizontal bar gauge proportional to RPM_MAX.

03 — Future scope
Where this project
can go next.

The hardware and firmware described here are a foundation. Each of these extensions adds a meaningful capability without requiring a major redesign.

/ 01

Wi-Fi data streaming

The ESP32's built-in Wi-Fi can push RPM readings to an MQTT broker, a web socket, or a REST endpoint. This enables real-time dashboards on a phone or PC with no additional hardware.

/ 02

Data logging to SD card

Adding an SPI-connected SD card module allows timestamped RPM logs to be written as CSV files and later analysed in Excel or Python for trend analysis and predictive maintenance.

/ 03

Tachometer alarm thresholds

A configurable over-speed alarm — using a buzzer or LED — can protect machinery from running above a safe RPM limit, turning this from a display device into an active safety instrument.

/ 04

Larger graphical display

Swapping to a 1.3-inch or 2.42-inch OLED, or a 2.8-inch TFT colour LCD, allows for an analogue needle gauge style display and plots of RPM history — more informative at a glance.

/ 05

Torque and power estimation

Combined with a current sensor on a motor supply, RPM and current readings can estimate shaft power (P = T x omega) for basic motor efficiency monitoring.

/ 06

Multi-axis measurement

A second encoder channel and a second ISR enable simultaneous monitoring of two independent shafts — useful for differential drive robots or dual-motor CNC systems.

/ 07

BLE to mobile app

The Bluetooth Low Energy stack in the ESP32 can broadcast RPM data to a custom mobile application, turning the phone into a wireless dashboard without requiring a Wi-Fi network.

/ 08

Closed-loop speed control

Feeding the RPM measurement back to a PID controller and a motor driver converts this from a passive meter into an active regulator that holds a target RPM regardless of load variation.

The codebase is intentionally transparent. Every constant has a comment. Every formula is written out. It is meant to be read, modified, and taken apart.
